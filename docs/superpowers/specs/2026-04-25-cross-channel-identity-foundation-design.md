# Cross-channel identity foundation (stages 1+2)

**Date:** 2026-04-25
**Status:** Approved

## Problem

The bot currently treats each channel as a separate auth domain with no shared identity:

- **Telegram** authorisation lives in the `allowed_chats` SQLite table; admin status is derived from the `ADMIN_CHAT_IDS` env var with a fallback to the first allowed chat.
- **Discord** authorisation reads `ALLOWED_DISCORD_USERS` directly from the env at every check; admin status reads `ADMIN_DISCORD_USERS`. No DB involvement.
- **WhatsApp** is the same pattern as Discord, with `ALLOWED_WHATSAPP_NUMBERS` / `ADMIN_WHATSAPP_NUMBERS`.

Three different storage models, three different fallback policies (Telegram has the "first allowed = admin" fallback, the other two don't), three different open-mode behaviours (only Telegram logs `warnOpenModeOnce`). Adding a fourth channel would mean copying the pattern for the fourth time. Linking the same physical person across channels is impossible ‚Äî they're three independent users to the bot.

## Goals

- One source of truth for authorisation: the database. Env-listed chat ids become first-boot seeds, not runtime authority.
- One auth-matrix policy applied uniformly to Telegram, Discord, and WhatsApp.
- A foundation that supports merging channels into a single logical user later, without re-doing the schema.
- No behavioural regression in production: the same chat ids stay authorised, the same admin stays admin.

## Non-goals (deferred to follow-up specs)

- **Memory by user_id.** Memory rows still index by `chat_id`. Migrating them to `user_id` is the next spec.
- **`/link` command.** Linking two existing chat-ids to the same user (e.g. Telegram + Discord = one person) is the spec after that.
- **Rate-limit by user_id.** Currently per-chat-id; per-user is a follow-up.
- **Removing `allowed_chats` table.** It stays as a backwards-compat shim, read through wrappers. A future spec drops it once nothing reads it.
- **Removing env-listed allowlists from runtime checks.** They become seeds-only, but the env vars still exist for migration purposes.

## Design

### Schema (migration v8)

```sql
CREATE TABLE users (
  user_id TEXT PRIMARY KEY,
  display_name TEXT,
  is_admin INTEGER NOT NULL DEFAULT 0,
  created_at INTEGER NOT NULL
);

CREATE TABLE user_chats (
  chat_id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
  channel TEXT NOT NULL CHECK(channel IN ('telegram','discord','whatsapp')),
  added_at INTEGER NOT NULL,
  added_by TEXT,
  note TEXT,
  last_seen_at INTEGER
);

CREATE INDEX idx_user_chats_user ON user_chats(user_id);
```

Both tables are created with `IF NOT EXISTS` so the migration is idempotent under the existing migration framework.

`user_id` format: literal prefix `u_` followed by 8 lowercase hex characters (`u_a3f1c2d4`). Generated by a helper in `users.ts`. The format is stable; if collision-free behaviour is ever needed at higher scale, the helper can switch to UUIDv4 without breaking consumers.

### Migration data move (inside v8 `up()`)

Order of operations inside the same transaction:

1. **Seed users from `allowed_chats`.** For every existing row (`chat_id`, `added_at`, `added_by`, `note`, `last_seen_at`):
   - Generate a fresh `user_id`. `display_name = chat_id`. `is_admin = 0`. `created_at = added_at`.
   - Insert into `users`.
   - Insert into `user_chats` with `channel = 'telegram'`, copying `added_at`, `added_by`, `note`, `last_seen_at`.

2. **Seed Discord from env.** For every id in `ALLOWED_DISCORD_USERS`:
   - `chatId = 'discord:' + rawId`. Skip if already present in `user_chats`.
   - Otherwise create user (`display_name = chatId`, `is_admin = 0`, `created_at = now`) and user_chats row (`channel = 'discord'`, `added_by = 'env'`, `note = 'seeded from ALLOWED_DISCORD_USERS'`).

3. **Seed WhatsApp from env.** Same pattern, `chatId = '<number>@s.whatsapp.net'`, channel `whatsapp`. (We use the bare-number form documented in the env, expand to JID inside the migration.)

4. **Apply admin flags from env.** For every id in `ADMIN_CHAT_IDS`, `ADMIN_DISCORD_USERS` (with `discord:` prefix), and `ADMIN_WHATSAPP_NUMBERS` (with `@s.whatsapp.net` suffix):
   - Find the matching user_chats row, set `users.is_admin = 1` for the corresponding `user_id`.
   - If no match (env points at a chat that wasn't seeded), skip silently. Logging at warn would be useful but the migration must not throw.

5. **Bootstrap admin fallback.** After step 4, if `users` has rows but no row with `is_admin = 1`, set `is_admin = 1` for the user with the smallest `created_at` (oldest first). This mirrors the existing Telegram-only fallback (`ADMIN_CHAT_IDS = ALLOWED_CHAT_IDS.slice(0, 1)`) and extends it to all channels.

If `allowed_chats` is empty AND all env-listed allowlists are empty, the migration leaves `users` empty. That's "open mode" ‚Äî auto-seeded on first incoming message (see Open mode section).

### Channel parser

There's already `src/channel-router.ts` with `classifyChatId(chatId): 'telegram' | 'discord' | 'whatsapp' | 'unknown'`. Reuse it. No new module.

For migration v8 we need to construct chat_ids from raw env values, which is a different operation:

```typescript
// src/channel-router.ts (extend the existing module)
export function discordChatId(rawUserId: string): string {
  return `discord:${rawUserId}`
}

export function whatsappChatId(rawNumber: string): string {
  return rawNumber.includes('@') ? rawNumber : `${rawNumber}@s.whatsapp.net`
}
```

Two thin helpers, no new abstraction.

### `src/users.ts` ‚Äî new module

Owns all reads and writes against `users` + `user_chats`. Public API:

```typescript
export interface UserLookup {
  userId: string
  isAdmin: boolean
  channel: ChannelKind
}

export function generateUserId(): string  // 'u_' + 8 lowercase hex

export function getUserByChat(chatId: string): UserLookup | null

export function countUsers(): number
export function isOpenMode(): boolean   // countUsers() === 0
export function isAuthorisedChat(chatId: string): boolean
// open mode ‚Üí true (and the caller should auto-add)
// non-empty users ‚Üí membership in user_chats

export function isAdminChat(chatId: string): boolean
// non-admin if not authorised; admin if user_chats row maps to a user with is_admin=1

export interface AddUserChatOpts {
  chatId: string
  channel: ChannelKind
  existingUserId?: string  // link to an existing user
  isAdmin?: boolean        // admin flag for the user (created or existing)
  addedBy?: string
  note?: string
  displayName?: string
}

export function addUserChat(opts: AddUserChatOpts): { userId: string; created: boolean }
// created=true if a new user was minted; false if linked to existing.
// Throws if chatId is already in user_chats and existingUserId mismatches.

export function removeUserChat(chatId: string): {
  removed: boolean
  userDeleted: boolean
  memoriesDeleted: number
  tasksDeleted: number
  sessionCleared: boolean
  preferencesCleared: boolean
}
// Removes chat row. If user has no more chats, deletes the user (CASCADE).
// Cleans memories, scheduled_tasks, sessions, chat_preferences for this chat_id
// (matching the existing removeAllowedChat semantics).

export function touchUserChat(chatId: string): void
// Updates last_seen_at for the chat. Replaces the existing touchAllowedChat.
```

Validation: `addUserChat` enforces channel/format consistency via `classifyChatId` (the chat_id must match the declared channel). `parseChatId` would throw on `unknown` ‚Äî `addUserChat` rejects with a clear error.

Bootstrap admin behaviour: when `addUserChat` adds the **first** user (current `countUsers() === 0`), if `isAdmin` is not explicitly false, it auto-promotes that user to admin. This applies to both env-seeded migration cases (handled inside the migration directly) and to runtime auto-add in open mode.

### Wrappers in `src/db.ts`

The existing exports keep their signatures ‚Äî callers across the codebase don't break. Their bodies are now thin wrappers over `users.ts`:

```typescript
// Existing: addAllowedChat(chatId, addedBy, note) ‚Äî Telegram-only
export function addAllowedChat(chatId: string, addedBy: string | null, note: string | null): boolean {
  if (!isValidTelegramChatId(chatId)) {
    throw new Error(`addAllowedChat: not a Telegram chat id: ${chatId.slice(0, 80)}`)
  }
  const result = users.addUserChat({
    chatId,
    channel: 'telegram',
    addedBy: addedBy ?? undefined,
    note: note ?? undefined,
  })
  return result.created || true  // legacy semantics: true if newly inserted
}

export function isAuthorised(chatId: number | string): boolean {
  return users.isAuthorisedChat(String(chatId))
}

export function isOpenMode(): boolean {
  return users.isOpenMode()
}

export function removeAllowedChat(chatId: string): RemoveChatResult {
  const r = users.removeUserChat(chatId)
  return {
    removed: r.removed,
    memoriesDeleted: r.memoriesDeleted,
    tasksDeleted: r.tasksDeleted,
    sessionCleared: r.sessionCleared,
    preferencesCleared: r.preferencesCleared,
  }
}

export function touchAllowedChat(chatId: string): void {
  users.touchUserChat(chatId)
}

export function listAllowedChats(): AllowedChatRow[] {
  // Project user_chats join users back into the legacy shape so /users command keeps working.
  // ...query joining user_chats LEFT JOIN users on user_id, returning AllowedChatRow shape.
}

export function isChatAllowed(chatId: string): boolean {
  return users.getUserByChat(chatId) !== null
}

export function countAllowedChats(): number {
  // Number of *chats*, not users. Existing callers use this to detect open mode.
  const row = getDb().prepare('SELECT COUNT(*) AS c FROM user_chats').get() as { c: number }
  return row.c
}
```

`seedAllowedChatsFromEnv` is removed ‚Äî its job is done at migration time now. Any test or caller of it gets updated to call `users.addUserChat` directly.

### Wrappers in `src/config.ts`

```typescript
// Existing public API stays the same:

}
export function isDiscordUserAuthorised(userId: string): boolean {
  return users.isAuthorisedChat(discordChatId(userId))
}

  return users.isAdminChat(discordChatId(userId))
}
export function isWhatsAppAuthorised(number: string): boolean {
  return users.isAuthorisedChat(whatsappChatId(number))
}

  return users.isAdminChat(whatsappChatId(number))

```

The corresponding `*Of` test-helpers (`isAdminOf`, `isDiscordUserAuthorisedOf`, etc.) ‚Äî these took an explicit allowlist for unit testing. They lose their meaning under DB-backed auth. We delete them. Tests that used them get rewritten to set up `users` + `user_chats` instead.

`ALLOWED_*` and `ADMIN_*` env exports stay in `config.ts` (still parsed at module load) but are unused in production. The migration imports them. A comment marks them as "first-boot seeds only".

### Open mode + auto-add

Current behaviour:
- Telegram: empty `allowed_chats` ‚Üí all chats authorised. First time a chat sends ‚Üí `warnOpenModeOnce` logs warn.
- Discord: empty `ALLOWED_DISCORD_USERS` ‚Üí all users authorised, no warn.
- WhatsApp: same as Discord.

Unified behaviour:
- `users` table empty (`isOpenMode() === true`) ‚Üí all chats authorised. First incoming message in a chat:
  - Logs warn once per process (per `chat_id`) via the existing `warnOpenModeOnce` helper, which we lift to a shared module so all three handlers use it.
  - Calls `users.addUserChat({ chatId, channel })` with no `existingUserId`. Auto-promotes to admin (bootstrap).
- After auto-add, `users` is no longer empty ‚Üí next inbound chat is rejected unless explicitly added.

Telegram's `seedAllowedChatsFromEnv` used to populate `allowed_chats` on first start ‚Äî that path is gone. Migration v8 does the seeding once.

### `/adduser` command

Existing command is in `src/commands/users.ts` (or wherever `/adduser` lives ‚Äî verify at implementation). Update to:

- Accept a chat_id in any format: Telegram numeric, `discord:<id>`, `<number>@s.whatsapp.net`. `parseChatId`-equivalent (reuse `classifyChatId`) determines the channel.
- Reject if format is unknown.
- Optional flags:
  - `--admin` ‚Äî set admin on the user.
  - `--user-id <existing>` ‚Äî link this chat to an existing user instead of creating a new one. (Foundation for the future `/link` flow; available now if you need to manually merge.)
  - `--note <text>` ‚Äî passes through to `user_chats.note`.

Existing `/users` listing command keeps working through the `listAllowedChats` wrapper; output may need a slight update to include channel and user_id, but the basic shape (one row per chat) is unchanged.

### Removal flow

`/removeuser <chat_id>` (existing) calls the wrapper `removeAllowedChat`, which now goes through `users.removeUserChat`. Behaviour change: removing the last chat of a user also removes the user. Removing a chat from a user with multiple chats leaves the user. We log this transition at info.

`removeAllowedChat` previously hard-rejected non-Telegram chat_ids via `isValidTelegramChatId`. We relax that ‚Äî `removeUserChat` accepts any chat_id format. The Telegram validation was a guard against a specific historical bug (env seeds polluted with WhatsApp JIDs). With the unified store, we trust `parseChatId`.

## Observability

- New log on user creation: `info { userId, channel, chatId, addedBy }, 'user created'`.
- New log on user deletion (chain reaction from chat removal): `info { userId }, 'user removed (last chat deleted)'`.
- Bootstrap admin promotion (first user becoming admin): `warn { userId }, 'bootstrap admin: promoted oldest user'`. (Both at migration time and at runtime open-mode auto-add.)
- Open-mode warn pattern: keep `warnOpenModeOnce` as-is, just shared between handlers.

No new metric events.

## Rollout

Single PR, single deploy. Migration v8 runs automatically on first start after upgrade.

Operator pre-deploy checklist:
- Verify `ADMIN_CHAT_IDS` (or whatever admin envs are set) point at the right user. The migration trusts these. If empty, bootstrap fallback picks the oldest user ‚Äî that may not be the operator. **Recommendation:** set `ADMIN_CHAT_IDS=<your telegram chat>` explicitly before deploying.
- No new env var to set. `WHATSAPP_AUTH_ENCRYPTION_KEY` from the previous spec is still required for Baileys, unchanged.

Post-deploy verification:
- `/status` ‚Äî should show admin flag if you're admin.
- `sqlite3 store/claudeclaw.db "SELECT user_id, display_name, is_admin FROM users"` ‚Äî sanity check.
- `/users` ‚Äî output should be unchanged (legacy wrapper preserves shape).

## Risks

- **Bootstrap admin landing on the wrong user.** If admin env vars are empty and the seed order makes a non-operator the oldest user, that user becomes admin. Mitigation: set `ADMIN_CHAT_IDS` explicitly before deploy. Test #1 covers the fallback case.
- **`removeAllowedChat` semantic change.** Previously it just removed the chat; now it may also delete the user. If a downstream consumer assumed user persistence after chat removal, it breaks. Mitigation: search for all callers in the implementation step. Currently only `/removeuser` command and one cleanup path in tests use it.
- **`countAllowedChats` semantic.** It still counts *chats*, not users. Open-mode detection (`isOpenMode()`) uses *users*. Two different counts. The wrapper documents this difference.
- **Test rewrites.** A handful of existing tests (`permissions.test.ts`, etc.) used pure-function `*Of` helpers that are gone. They need DB setup. The implementation plan budgets one task for this rewrite.
- **Legacy `seedAllowedChatsFromEnv`.** The function disappears. Any test or script that called it directly breaks. Implementation grep confirms callers; the only one is `index.ts` startup, which the migration replaces.

## Open questions resolved during design

- **Why not just merge `db.ts`?** It's already 700+ lines and crossing several concerns. Splitting users management into its own file keeps both files focused.
- **Why not UUIDv4 for user_id?** Readability. `u_a3f1c2d4` fits in CLI args without quoting and is human-distinguishable from chat ids.
- **Why both `users` and `user_chats` instead of one denormalised table?** Many-to-one mapping is the whole point of this design ‚Äî one user, several chats. A single-table design would force the mapping into a JSON blob.
